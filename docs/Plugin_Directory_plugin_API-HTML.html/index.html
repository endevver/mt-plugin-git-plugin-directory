Plugin Directory plugin APIs
Introduction
This document describes the resources that make up the Plugin Directory plugin API v1. This API actually consists of two distinct APIs:
Repository metadata API
Provides plugin authors control over nearly every aspect of their plugin's display in the plugin directory all from the comfort of their working Git repository and favorite text editor.
REST API
This API provides a RESTful, HTTP-based interface for programmatic access to all major system functions creating numerous opportunities for integration with third-party applications and utilities including, perhaps most importantly, Git hook scripts for automating Plugin Directory updates.
The Repository Metadata API
What is the Repository Metadata API
The Metadata API is the language and process plugin authors use to richly describe their plugins to the Plugin Directory. This is the very heart of the Plugin Directory and was designed to combat the two biggest problems with the original MT plugin directory (and, really, any user-generated directory):
Barrier to entry/Friction of interaction
Software development is hard enough without adding extra steps like going to the Plugin Directory, logging in, filling out some huge form to submit a plugin you just released or updating the same form every time you push out an update.
Data rot
As time goes on, information not constantly tended to becomes stale and invalid.
How the API solves the problem
The Metadata API serves to eliminate (to the greatest possible degree) the overhead of submitting and updating plugin information by adapting to and integrating tightly with the typical workflow of a plugin developer. Instead of forcing plugin developers to constantly cater to the maintenance needs of an external, centralized directory, the design leverages the power and decentralized nature of Git to automate nearly every aspect of plugin directory maintenance and respect the true canonical source for a plugin's metadata: each author's working repository.
Using the Metadata API
The vast majority of plugin metadata displayed by the Plugin Directory is provided and directly controlled by two YAML files located at the root of each plugin's repository: config.yaml and plugin.yaml.
config.yaml—MT API interactions and plugin behaviors
The config.yaml file is already well established amongst plugin developers as the file which defines a plugin's basic information as well as the entirety of its behaviors and interactions with MT/Melody's plugin API. Other plugins such as Config Assistant and Theme Manager extend the configuration capabilities even further.
Required fields
The Plugin Directory does not impose any changes on the vocabulary used or functionality provided by the config.yaml. However, to be included into the Plugin Directory, the following fields must be defined.
name
The name of the plugin as shown within MT/Melody in the listing of installed plugins.
id
A string identifier for the plugin that should be short (< ~20 chars) but unique across *all* plugins. By convention, they are usually camel-cased and have no punctuation such as dashes, underscores, periods or colons. Examples: ConfigAssistant, ThemeManager, OrderedCF, Log4MT
version
The plugin's version number. We generally suggest a three-part version, such as 1.0.0.
plugin.yaml—Plugin meta-information
The plugin.yaml file is newly introduced by this specification and intended to contain all of the necessary metadata not already contained in the config.yaml. By putting this extra information in a separate file only consumed by the Plugin Directory, it serves as a performance-friendly extension of each plugin's config.yaml file.
Required fields
config_yaml_path
The filesystem path to the config.yaml, relative to plugin.yaml. Example: plugins/Foo/config.yaml
This field is only required if config.yaml is not in the root of the repository
version_tag
The Git tag corresponding to the latest production-ready release of the plugin.
This field is only required if the version specified in your config.yaml does not exactly match the tag OR if you also have tags corresponding subsequent beta-quality releases.
version_tag_beta
The Git tag corresponding to the latest beta-quality release of the plugin.
This field is only required if you have tagged a beta-quality release which you wish to advertise on the Plugin Directory and the version specified in your config.yaml does not exactly match the tag.
Strongly suggested fields
license
license_url
version_compatibility_mt
version_compatibility_melody
latest_dev_branch
The Git branch representing the latest development for your plugin. This will provide the Plugin Directory with the pulse of the activity in a repo and also provide users with the ability to download the very latest, cutting edge code.
bug_tracker_url
Optional fields
name
This field can be used to override the value of the identically-named key in config.yaml.

Plugin author methods
The following methods are all functions commonly used by plugin authors who wish to add new plugins to the centralized plugin directory or manage previously submitted ones. All require authentication by the plugin author, the Plugin Directory blog administrator or a system administrator.
ADD
Submits a Git repository URL into the system for inclusion in the directory,
pending validation and moderation.
Authentication required: Yes
Authorized roles: Plugin owner, blog administrator or system administrator
Required parameters
git_url
The read-only Git clone URL for the repository being submitted.
username
The plugin author's MT/Melody username
token
The plugin author's Plugin Directory API token which can be found on their user profile.
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
A numeric record ID that uniquely identifies the plugin and can be used as a reference to it in other calls. Unless the record could not be created due to some database issue, this key/value will be included in the response even if the submission is deemed invalid.
basename
A dirified string that uniquely identifies the plugin and can be used as a reference to it in other calls. Unless the record could not be created due to some database issue, this key/value will be included in the response even if the submission is deemed invalid.
validated
A numeric, boolean flag (0 or 1) indicating whether the submission has been checked by the validation process. If the system is configured to process submission synchronously with each request, this will return true. If asynchronous processing is enabled, a false value will be returned and, as with all status changes, the results of the validation will be sent to the email address listed in the plugin author's user profile once the process has been completed.
error
A human-friendly message that describes an error condition. This key/value will only be present when an error has occurred and will be accompanied by an HTTP error code sent in the response headers. Hence, the lack of a value for this key combined with a true value for the validated key, indicates a successful and complete submission.
EDIT
This method is nearly identical to the "ADD" method but is used to update an existing record rather than create a new one. This might be necessary if your initial submission contained an incorrect URL, you rename or transfer ownership of your plugin thereby altering the clone URL or if you migrate your repository to an entirely new provider. The Git URL and repository will undergo re-validation (as described in the documentation for the ADD method) prior to acceptance and publishing.
Authentication required: Yes
Authorized roles: Plugin owner, blog administrator or system administrator
Required parameters
The only difference from the "ADD" method is that the request contains an "id" or "basename" parameter used to identify an existing record to be modified.
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
git_url
The new Git clone URL for the canonical repository
username
The plugin author's MT/Melody username
token
The plugin author's Plugin Directory API token which can be found on their user profile.
Return value
Please see ADD method return values which are identical to this method's
REMOVE
This method allows a plugin author to delete a repository's record from the system, thereby removing the plugin from the plugin directory.
Authentication required: Yes
Authorized roles: Plugin owner, blog administrator or system administrator
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
username
The plugin author's MT/Melody username
token
The plugin author's Plugin Directory API token which can be found on their user profile.
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
The no-longer-valid ID of the removed record
basename
The no-longer-valid basename of the removed record
git_url
The git clone URL contained in the removed record
error
A human-friendly message that describes an error condition. This key/value will only be present when an error has occurred and will be accompanied by an HTTP error code sent in the response headers.
Plugin directory moderator methods
The following methods require authentication by a user with either a Moderator or Blog administrator role on the blog or a System Administrator.
APPROVE
This method changes a plugin's moderation status signaling acceptance and triggers the republishing actions needed to display the plugin in the published directory.
Authentication required: Yes
Authorized roles: Blog moderator, blog administrator or system administrator
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
username
The moderator's MT/Melody username
token
The moderator's Plugin Directory API token which can be found on their user profile.
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
The numeric ID of the plugin's record
basename
The basename string of the plugin's record
moderation_status
A boolean flag (0 or 1) indicating whether the submission is rejected or approved, respectively. An undefined value indicates that moderation is still pending.
error
A human-friendly message that describes an error condition. This key/value will only be present when an error has occurred and will be accompanied by an HTTP error code sent in the response headers.
REJECT
This method changes a plugin's moderation status signaling rejection and, if necessary, triggers republishing actions needed to remove the plugin from the published directory.
Authentication required: Yes
Authorized roles: Blog moderator, blog administrator or system administrator
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
username
The moderator's MT/Melody username
token
The moderator's Plugin Directory API token which can be found on their user profile.
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
The numeric ID of the plugin's record
basename
The basename string of the plugin's record
moderation_status
A boolean flag (0 or 1) indicating whether the submission is rejected or approved, respectively. An undefined value indicates that moderation is still pending.
error
A human-friendly message that describes an error condition. This key/value will only be present when an error has occurred and will be accompanied by an HTTP error code sent in the response headers.
Internal methods
Internal methods are usually called only by internal code but can be initiated by the Plugin Directory blog administrators for testing and queue clearing purposes.
VALIDATE
Performs a series of syntax and packaging checks on a repository looking for
things like invalid or unparse-able YAML files, unsupported packaging formats
and the like
Authentication required: Yes
Authorized roles: Blog administrator or system administrator
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
username
The blog admin's MT/Melody username
token
The blog admin's Plugin Directory API token which can be found on their user profile.
Optional parameters
asynchronous
A numeric boolean value (0 or 1) which specifies whether the validation process should be done synchronously to the request or if it should be submitted as a job for later processing. Since this defaults to 0, it's only necessary if you wish to force an asynchronous process.
Return value
Please see ADD method return values which are identical to this method's
NOTIFY
Authentication required: Yes
Authorized roles: Blog administrator or system administrator
Public methods
PING
This method notifies the system of a potential update to the tracked
repository. In response, the system will asynchronously initiate a pull from
the repository and evaluate the metadata to determine whether a directory
update is needed.
Authentication required: No
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
The numeric ID of the plugin's record
basename
The basename string of the plugin's record
modified_on
A timestamp indicating the when the record was last modified
error
A human-friendly message that describes an error condition. This key/value will only be present when an error has occurred and will be accompanied by an HTTP error code sent in the response headers.
SHOW
This method provides access to some or all of the details of a plugin's record, depending on whether or not the request is authenticated using the plugin author's credentials.
Authentication required: Only for more detailed views
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
Optional parameters
username
A MT/Melody username
token
The Plugin Directory API token corresponding to the specified username
Return value
This method returns a JSON-formatted struct containing one or more of the following keys, depending on whether the request was authenticated and the role of the authenticated user.
