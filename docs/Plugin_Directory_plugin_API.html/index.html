Plugin Directory plugin APIs
Introduction
About the Plugin Directory API
This document describes the resources that make up the Plugin Directory plugin API v1, which is, itself, comprised of two distinct APIs:
Repository metadata API
Provides plugin authors control over nearly every aspect of their plugin's display in the plugin directory all from the comfort of their working Git repository and favorite text editor.
REST API
Provides a RESTful, HTTP-based interface for programmatic access to all major system functions creating numerous opportunities for integration with third-party applications and utilities including, perhaps most importantly, Git hook scripts for automating Plugin Directory updates.
Goals of the Plugin Directory API
The primary goals of both APIs are as follows:
A frictionless user experience for the *content creators*: **the plugin developers**
Software development is difficult enough without burdening developers with unnecessary extra requirements, processes and tasks tacked onto the end of every software release. To maximize the Plugin Directory's success, we strive to make the user experience for developer's one that yields the maximum benefit for the lowest possible effort.
A rich and rewarding user experience for *content consumers*: **users seeking solutions to their problems**
End users, administrators, et al. visit a plugin directory for one primary reason: To find a solid solution to an existing problem as quickly and efficiently as possible. Like everything on the internet, the difficulty is not in finding solutions but separating the wheat from the chaff to identify the *best* solution. To maximize the Plugin Directory's success, we focus on facilitating and streamlining that process making the user experience for developer's one that yields the maximum benefit for the lowest possible effort.
A drop-in and fully-extensible solution for anyone wanting to create and maintain a "next generation" MT/Melody plugin directory
A enable, facilitate and promote the creativity of the community and unimagined use cases and maximize the potential for programmatic interaction with the Plugin Directory by third-party applications and utilities.
Tactics for a frictionless UX for creators
One-click plugin submission
For registered and authenticated plugin developers, submitting a plugin to the directory should be as simple as providing a single URL pointing to the canonical source of your plugin and all of its metadata.
Do I hear zero-clicks?
Why put the onus and burden of submitting a publicly available plugin entirely on the plugin author, thereby creating a bottleneck in the very process we wish to make frictionless? Have we learned nothing from Wikipedia? There are so many helpful and knowledgeable people on the internet who have also mastered the ancient skils of "copy, paste and click". The Plugin Directory should welcome their contributions with open arms and promote the collaborative, "barn raising" spirit of its community.
Work with me here!
Most centralized directories, systems and services require you to spend precious time retyping the same things, clicking the same boxes, re-friending the same people, reorganizing the same data again and again. We strive to follow a different example creating a system modeled after one of the most sacred developer mantras: "Don't Repeat Yourself". What does this mean?
The directory should fully leverage a developer's past efforts by acquiring information about a plugin from the canonical sources.
A directory with the intelligence to interact seamlessly with the developer's most important tool—the version control system—provides for automatic and immediate discovery of new releases
Respect and fully leverage canonical sources of metadata and information
Programmatic access
If there's one thing every developer loves, it's an API and the Plugin Directory will have two powerful ones right out of the gate. Why should a developer spend time submitting all of his previously created plugins to a directory individually when he can write a simple Perl or shell script to do the whole thing at once?
Passive, automated updates
Endow the system with the intelligence to seamlessly interact with a plugin developer's most important tool—the version control system—making the process of updating the plugin directory a completely passive and automated one.
To significantly enhance the user experience of plugin developers by eliminating, to the greatest degree possible, the barriers to entry and overhead of submitting and updating Plugin Directory listings for their plugins
A directory with the intelligence to interact seamlessly with the developer's most important tool—the version control system—provides for and a heartbeat monitor for development activity and the health of the software project
Tactics for a rich, rewarding UX for consumers
leveraging existing plugin metadata from the config.yaml which defines the entirety of a plugin's programmatic behaviors and MT/Melody API usage
Providing users with a rich source of new metadata whose intended audience is humans, specifically current and potential users of a plugin
providing a more complete picture of each plugin's current and past development activity allowing users to more easily find actively developed plugins
No data rot
As time goes on, information not constantly tended to becomes stale and invalid.
The Repository Metadata API
What is the Repository Metadata API
The Metadata API is the language and set of supporting processes that plugin authors use to richly describe their plugins to the Plugin Directory.
How the API solves the problem
The Metadata API serves to eliminate (to the greatest possible degree) the overhead of submitting and updating plugin information by adapting to and integrating tightly with the typical workflow of a plugin developer.

Instead of forcing plugin developers to constantly cater to the maintenance needs of an external, centralized directory, the design leverages the power and decentralized nature of Git to automate nearly every aspect of plugin directory maintenance and respect the true canonical source for a plugin's metadata: each author's working repository.
Using the Metadata API
The vast majority of plugin metadata contained in and displayed by the Plugin Directory is provided and directly controlled by two YAML files located in each plugin's repository: config.yaml and plugin.yaml.
About the YAML metadata files
config.yaml—MT API interactions and plugin behaviors
The config.yaml, introduced in MT 4.0, is a file located in the root of a plugin's envelope which contains all of the plugin's basic information as well as definitions for the entirety of its behaviors and interactions with MT/Melody's plugin API. Other plugins such as Config Assistant and Theme Manager extend the configuration capabilities even further. See "Appendix A: `config.yaml` reference" for full details.
plugin.yaml—Plugin meta-information for humans
The plugin.yaml is a file located in the root of a plugin repository that supplements the `config.yaml` metadata (which is primarily intended for consumption by the code of the currently executing application) with information primarily of interest to *humans*—more specifically, current and potential future users of the plugin.

Just a few examples of `plugin.yaml` metadata are listed below. See "Appendix B: `plugin.yaml` reference" for full details.
URLs or email addresses for user support, feedback and/or a case tracking tool for the plugin
The version numbers (actually Git tag) for the latest production-ready and beta-quality releases, if any
A long, HTML- or Markdown-formatted description of the plugin suitable for display on the plugin's detail page
Declarations of compatibility (or lack thereof) with specific MT and Melody versions
Processing of YAML metadata
Together, the `config.yaml` and `plugin.yaml` metadata files act as the Plugin Directory's orientation guide to the plugin's source code and intended behavior and is these two files that are consumed and processed first and foremost any time the Plugin Directory's scans the repository.
What triggers a repository scan?
Submission of a plugin to the directory
An update to a plugin's record modifying the Git repository URL
Receipt of a plugin update ping via the REST API
Asynchronous operations
Worker Daemons
All processing of new and updated plugin repositories is executed asynchronously by one of two worker daemons
Validator daemon
Updater daemon
File processing order
Developer notification of status changes

Success or failure
There are two return value keys that are special in that one (and only one) of them will always be included in every single JSON struct returned by the application. These two keys indicate either some level of success or failure for the previous request.
status
A word or short phrase indicating the status of the last request. This key/value will only be present when the last request ran without error and will always be accompanied by an HTTP response code sent in the response headers, most likely in the 2xx or 3xx range.
error
A human-friendly message that describes an error condition in the previous request. This key/value will only be present when an error has occurred and will be accompanied by an HTTP response code sent in the response headers, most likely in the 4xx or 5xx range.

Plugin author methods
The following methods are all functions commonly used by plugin authors who wish to add new plugins to the centralized plugin directory or manage previously submitted ones. All require authentication by the plugin author, the Plugin Directory blog administrator or a system administrator.
POST /plugin
Submits a plugin (represented in the request by a simple Git repository URL) to the system for inclusion in the directory, pending validation and moderation.
Authentication required: Yes
Authorized roles: Plugin owner, blog administrator or system administrator
Required parameters
git_url
The read-only Git clone URL for the repository being submitted.
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
A numeric record ID that uniquely identifies the plugin and can be used as a reference to it in other calls. Unless the record could not be created due to some database issue, this key/value will be included in the response even if the submission is deemed invalid.
basename
A dirified string that uniquely identifies the plugin and can be used as a reference to it in other calls. Unless the record could not be created due to some database issue, this key/value will be included in the response even if the submission is deemed invalid.
validated
A numeric, boolean flag (0 or 1) indicating whether the submission has been checked by the validation process. If the system is configured to process submission synchronously with each request, this will return true. If asynchronous processing is enabled, a false value will be returned and, as with all status changes, the results of the validation will be sent to the email address listed in the plugin author's user profile once the process has been completed.
HTTP has a specific response code, 202 (Accepted), the meaning of which is defined as “The request has been accepted for processing, but the processing has not been completed.” This is obviously exactly what we’re looking for. Regarding the result, there are multiple options: The server can return a URI of a resource which the client can GET to access the result (although if it has been created specifically due to this request, a 201 Created would probably be better). Or the client can include a URI that it expects the server to POST the result to once it’s done
POST /plugin/:ID
Updates a single field of the plugin record specified by the ID parameter. The field can be specified either as an extension to the URL (e.g. `POST /plugin/123/mod_status/approved`) or as a parameter in the POST payload. The latter would be a better choice for any fields with more complex values like the `git_url` field

PUT /plugin/:ID
This method is nearly identical to the "ADD" method but is used to replace an existing record rather than create a new one. This might be necessary if your initial submission contained an incorrect URL, you rename or transfer ownership of your plugin thereby altering the clone URL or if you migrate your repository to an entirely new provider. The Git URL and repository will undergo re-validation (as described in the documentation for the ADD method) prior to acceptance and publishing.
Authentication required: Yes
Authorized roles: Plugin owner, blog administrator or system administrator
Required parameters
The only difference from the "ADD" method is that the request contains an "id" or "basename" parameter used to identify an existing record to be modified.
git_url
The new Git clone URL for the canonical repository
Return value
Please see ADD method return values which are identical to this method's
DELETE /plugin/:ID
Deletes the record of a plugin whose ID matches that specified by the ID parameter.
Authentication required: Yes
Authorized roles: Plugin owner, blog administrator or system administrator
Required parameters
id or basename
Requests may use either the numeric id or basename string to refer to the plugin of record
Return value
This method returns a JSON-formatted struct containing one or more of the following keys.
id
The no-longer-valid ID of the removed record
basename
The no-longer-valid basename of the removed record
git_url
The git clone URL contained in the removed record

Appendix A: `config.yaml` reference
Location of file: Root directory of the repository (recommended)
We strongly suggest putting your `config.yaml` (and all other co-located resources such as `docs/`, `extlib/`, `lib/`, `tmpl/`, `static/`, `t`, `tools`, etc) in the **root directory of your repository** such that it corresponds with the *plugin envelope* (i.e. the folder you install in the `plugins/` or `addons/` directory). This is a departure from the traditional convention of structuring your plugin folder hierarchy from the perspective of the `MT_HOME` directory.
Reasons you should be using a plugin envelope-style layout
There are many excellent reasons and developer- and end-user benefits in using a repository layout that corresponds to the plugin envelope:
Significantly simplifies plugin installation
Eliminates two layers of folder nesting which are a constant source of confusion for less saavy users and additionally a navigational/file management irritation for most developers
Foo/plugins/Foo and Foo/addons/Foo become simply Foo
NOTE: This requires the automated static file deployment feature of Configuration Assistant which is available for MT and bundled by default in Melody
Installing/updating from archive is dead simple
`unzip Foo.zip; mv Foo plugins/`
Installing/updating via git is dead simple
`cd plugins/; git clone GITURL;` or a git pull from within the plugin envelope
Keeps important meta-files and other resources together and updated with the source code
Examples: README, LICENSE and CHANGES files as well as helper scripts (`tools/`), unit tests (`t/`) and documentation (`docs/`)
Enables full use of **git submodules** for combining code from multiple repositories into a single repository
Example: Including a Git managed CPAN module in your `extlib` directory
Developers can easily include the contents of external Git-managed libraries (e.g. CPAN modules) directly into their own repository (e.g. in `extlib/`)
Administrators who use Git as version control for their entire MT installation can easily add plugins like this as submodules in the addons/plugins directory
It makes programmatic discovery of your config.yaml much simpler and faster
For example: By the Plugin Directory or a local script created to assist in searching your many plugin repositories
If you must use the legacy `MT_HOME`-style layout
If you cannot use the plugin envelope-style layout, you must specify the root relative path to your `config.yaml` in the `plugin.yaml` file using the `config_yaml_path` key described later in this section..
Required fields
The Plugin Directory does not impose any changes to the syntax or supported vocabulary of the config.yaml file. However, the Plugin Directory requires the following fields to be defined:
`name`
The name of the plugin as shown within MT/Melody in the listing of installed plugins.
This can be overridden in the plugin.yaml if, for some reason, you want a different display name in the Plugin Directory
`id`
A string identifier for the plugin that should be short (< ~20 chars) but unique across *all* plugins. By convention, they are usually camel-cased and have no punctuation such as dashes, underscores, periods or colons. Examples: ConfigAssistant, ThemeManager, OrderedCF, Log4MT
Actually, I'm not convinced this is important, especially considering that it's entirely ignored by MT and maybe Melody in important methods like MT's `component()`. (Plugin envelope is used, WTF?)
`version`
The plugin's version number which is normally displayed on the MT/Melody plugin listing page.
Use dotted-integer tuples!
We strongly suggest strict adherence to using a tuple of dotted integers (such as 0.9.23, 1.0.0 and 2.1.24) where each part is an incrementing integer representing the version's major, minor and bug fix revision (respectively). This format balances the natural human need for semantic meaning with the unyielding computational need for reliable input in producing reliable output (e.g. version comparisons and manipulations).
Use the exact same string as your Git tag for each release
We also strongly suggest that you use the exact same string as your git tag corresponding to each versioned release of your plugin. This consistency helps reduce confusion for users, collaborators and the Plugin Directory itself.
Appendix B: `plugin.yaml` reference
plugin.yaml—Plugin meta-information
The plugin.yaml file is newly introduced by this specification and intended to contain all of the necessary metadata not already contained in the config.yaml. By putting this extra information in a separate file only consumed by the Plugin Directory, it serves as a performance-friendly extension of each plugin's config.yaml file.
Required fields
config_yaml_path
The filesystem path to the config.yaml, relative to plugin.yaml. Example: plugins/Foo/config.yaml
This field is only required if config.yaml is not in the root of the repository
version_tag
The Git tag corresponding to the latest production-ready release of the plugin.
This field is only required if the version specified in your config.yaml does not exactly match the tag OR if you also have tags corresponding subsequent beta-quality releases.
version_tag_beta
The Git tag corresponding to the latest beta-quality release of the plugin.
This field is only required if you have tagged a beta-quality release which you wish to advertise on the Plugin Directory and the version specified in your config.yaml does not exactly match the tag.
Strongly suggested fields
long_description
A root-relative path to a file containing a longer description (HTML- or Markdown-formatted) of the plugin than that provided in `config.yaml`'s `description` field. This will be displayed prominently as the plugin's main description whereas the latter will be stored as an abstract displayed on search results and compact listing pages.
plugin_feed_url
A URL to an Atom or RSS formatted feed containing plugin-specific news and updates.
license
A unique code denoting your plugin's license.

license_url
A canonical URL for your plugin's license, e.g. www.gnu.org—gpl-2.0-standalone.html
version_compatibility_mt
A declaration of the plugin's MT version compatibility. What format should this take?
version_compatibility_melody
A declaration of the plugin's Melody version compatibility. What format should this take?
git_dev_branch
The Git branch representing the latest development for your plugin. This will provide the Plugin Directory with the ability to inform potential users of your plugin about the level of development activity and provide them with details of and downloads for the very latest, cutting edge code.
feedback_url
The URL to your plugin's case tracking tool (if it has one) for reporting bugs and requesting new features. Supports `mailto:` URLs for developers who prefer or require feedback submission via email.
support_url
A URL leading to human-powered support, for example a help ticket system, community forum, or even a mailto: URL for email-based support
NOTE: This is not a link to the documentation which should be specified using `doc_link` in `config.yaml`. This is for directing the user to actual humans.
Optional fields
name
This field can be used to override the value of the identically-named key in config.yaml.
